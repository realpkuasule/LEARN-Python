## 核心讲解

### 8.1 为什么要写函数——"每次攻击都重写一遍？"

没有函数的时候，每次攻击都要完整写一遍逻辑：

```python
# 没有函数——每次攻击都要写一遍完整逻辑
# 勇者攻击史莱姆
slime_hp = 30
slime_def = 2
hero_atk = 15
slime_hp = slime_hp - (hero_atk - slime_def)
print(f"勇者攻击了史莱姆！造成{hero_atk - slime_def}点伤害")

# 勇者攻击哥布林
goblin_hp = 50
goblin_def = 5
goblin_hp = goblin_hp - (hero_atk - goblin_def)
print(f"勇者攻击了哥布林！造成{hero_atk - goblin_def}点伤害")

# 重复几十次...迟早会写错
```

RPG 游戏里你不是每场战斗都重新设计技能——你选"攻击"，系统自动执行。函数让你给代码起名字，一次定义，到处使用：

```python
# 有函数——「攻击」就是一个技能名
def attack(attacker, defender):
    """攻击！attacker 和 defender 都是字典格式的属性卡"""
    damage = attacker["atk"] - defender["def"]
    if damage < 1:
        damage = 1   # 保底 1 点伤害
    defender["hp"] -= damage
    print(f"{attacker['name']} 攻击了 {defender['name']}！造成 {damage} 点伤害")

# 先创建角色和怪物的"属性卡"
hero = {"name": "勇者刘老三", "hp": 100, "atk": 15, "def": 5}
slime = {"name": "史莱姆", "hp": 30, "atk": 8, "def": 2}
goblin = {"name": "哥布林", "hp": 50, "atk": 12, "def": 5}

# 每次攻击一句话——像在游戏菜单里选指令！
attack(hero, slime)      # 勇者攻击史莱姆
attack(hero, goblin)     # 勇者攻击哥布林
attack(goblin, hero)     # 怪物也能用！哥布林反击
```

**逐行解读**：

- 第 2 行：`def attack(attacker, defender):`——定义函数，名字叫 `attack`，需要两个参数：攻击方和防守方
- 第 3 行：三引号字符串是函数的"帮助文档"——给队友和 AI 看的
- 第 4 行：伤害公式——攻击方的 atk 减去防守方的 def（所有角色统一用 `atk` 和 `def` 字段）
- 第 5-6 行：如果伤害小于 1（对方防御太高），保底 1 点——不能白打
- 第 7 行：从防守方 HP 里扣掉伤害——注意这是修改了字典里的值，所以外面的 `slime["hp"]` 也会变
- 第 8 行：打印战斗日志，用 f-string 嵌入名字和数字
- 第 11-13 行：创建角色属性卡——注意都在函数调用**之前**定义
- 第 16-18 行：调用函数——`attack(hero, slime)` 把 hero 传给 attacker、slime 传给 defender

**类比**：RPG 的技能系统。每个技能都是一段固定的动画 + 计算——你选"火焰斩"，系统自动播放动画、算伤害。你不会每场战斗都重新设计技能。函数就是你的技能设计器——`attack` 是基础攻击，后面你还能定义 `fire_slash`（火焰斩）、`heal`（治疗术）。

**核心理解**：`attack(hero, slime)` 被调用时，`attacker` 指向 `hero`，`defender` 指向 `slime`。所以 `defender["hp"] -= damage` 实际上修改的是 `slime["hp"]`——函数内外数据是连通的（因为字典是可变对象）。

### 8.2 def 定义函数——"把药水系统做成函数"

函数不只能封装攻击——任何可以命名的动作都可以写成函数。

```python
def use_potion(hero, potion_type="回复药"):
    """使用药水——默认是回复药"""
    # 药水效果表（字典套字典）
    potions = {
        "回复药": {"hp": 30, "mp": 0},
        "魔力药": {"hp": 0, "mp": 20},
        "完全回复药": {"hp": 9999, "mp": 9999},
    }
    effect = potions[potion_type]
    # 恢复生命——但不超过最大值
    hero["hp"] = min(hero["hp"] + effect["hp"], hero["max_hp"])
    hero["mp"] = min(hero["mp"] + effect["mp"], hero["max_mp"])
    print(f"使用了{potion_type}！HP+{effect['hp']} MP+{effect['mp']}")

# 创建勇者（升级后的属性）
hero = {"name": "勇者刘老三", "hp": 85, "max_hp": 100, "mp": 40, "max_mp": 50, "atk": 15, "def": 5}

# 在战斗中使用不同类型的药水
use_potion(hero, "回复药")   # → HP 恢复到 100（不会超过 max_hp）
use_potion(hero, "魔力药")   # → MP 恢复到 50（不会超过 max_mp）
use_potion(hero)             # → 不指定类型，默认用"回复药"
```

**逐行解读**：

- 第 1 行：`def use_potion(hero, potion_type="回复药"):`——第二个参数有默认值，不传就用"回复药"
- 第 4-8 行：药水效果表——用字典存每种药水的 HP/MP 恢复量
- 第 10-11 行：用 `min()` 防止恢复量超过最大值——比如当前 HP 85，回复药 +30 应该是 100，不是 115
- 第 14 行：创建 hero——注意多了 `max_hp` 和 `max_mp` 字段，等级高了上限也会提升
- 第 17 行：`use_potion(hero, "回复药")`——指定了药水类型
- 第 19 行：`use_potion(hero)`——没指定类型，走默认值"回复药"

**类比**：你把所有药水的效果做成了一张表（字典），用函数统一处理。新加一种药水？只需在表里加一行，函数不用改。这就是"数据驱动"的思想——函数逻辑不变，数据可变。

### 8.3 参数的类型与用法——"法师的魔法书"

函数参数有四种传法——就像 RPG 战斗菜单里的不同选择方式。

```python
def cast_spell(caster, target, spell_name="火球术", boosted=False):
    """施放魔法——默认是火球术，默认不强化"""
    # 魔法书——每个魔法的消耗和基础效果
    spells = {
        "火球术":  {"mp_cost": 10, "base_damage": 30},
        "暴风雪":  {"mp_cost": 25, "base_damage": 80},
        "治愈术":  {"mp_cost": 15, "base_heal": 50},
    }
    spell = spells[spell_name]

    # 检查 MP 够不够
    if caster["mp"] < spell["mp_cost"]:
        print(f"MP 不足！需要 {spell['mp_cost']}，当前只有 {caster['mp']}")
        return   # MP 不够，直接结束函数

    caster["mp"] -= spell["mp_cost"]

    # 伤害型魔法
    if "base_damage" in spell:
        damage = spell["base_damage"]
        if boosted:
            damage = damage * 1.5   # 魔力强化——伤害 ×1.5
        target["hp"] -= damage
        boost_text = "（魔力强化！）" if boosted else ""
        print(f"{caster['name']} 释放 {spell_name}！造成 {damage} 点伤害{boost_text}")

    # 治疗型魔法
    elif "base_heal" in spell:
        heal = spell["base_heal"]
        target["hp"] = min(target["hp"] + heal, target.get("max_hp", target["hp"] + heal))
        print(f"{caster['name']} 释放 {spell_name}！恢复 {heal} 点 HP")

# 创建法师和怪物
mage = {"name": "大法师", "hp": 60, "mp": 100, "atk": 5}
boss = {"name": "恶龙", "hp": 500, "atk": 80, "def": 30, "max_hp": 500}

# 四种调用方式——对应四种参数传法
cast_spell(mage, boss)                          # 1. 全用默认：火球术、不强化
cast_spell(mage, boss, "暴风雪")                 # 2. 指定魔法名（位置参数）
cast_spell(mage, boss, "火球术", True)           # 3. 按位置全传
cast_spell(mage, boss, boosted=True)             # 4. 关键字参数：跳过 spell_name，直接用默认火球 + 强化
```

**逐行解读**：

- 第 1 行：四个参数——`caster` 和 `target` 是必须的（位置参数），`spell_name` 和 `boosted` 有默认值（默认参数）
- 第 4-8 行：魔法书字典——每种魔法有不同的消耗和效果
- 第 11-13 行：MP 不够的处理——打印提示后 `return` 直接结束函数（不造成任何效果）
- 第 15 行：扣 MP——不管什么魔法都要消耗
- 第 18-22 行：伤害魔法分支——如果 spell 里有 `base_damage` 字段，走伤害逻辑
- 第 19-20 行：`boosted` 为 True 时伤害 ×1.5
- 第 25-27 行：治疗魔法分支——用 `min()` 防止超过最大生命值
- 第 31-36 行：四种调用方式展示参数的灵活性

**四种参数传法对比**：

| 传法       | 代码                                             | 说明                                                    |
| ---------- | ------------------------------------------------ | ------------------------------------------------------- |
| 位置参数   | `cast_spell(mage, boss, "暴风雪")`               | 按顺序传，mage→caster, boss→target, "暴风雪"→spell_name |
| 默认参数   | `cast_spell(mage, boss)`                         | spell_name 默认"火球术"，boosted 默认 False             |
| 关键字参数 | `cast_spell(mage, boss, boosted=True)`           | 跳过 spell_name 直接指定 boosted——不用按顺序            |
| 混合       | `cast_spell(mage, boss, "暴风雪", boosted=True)` | 位置参数在前，关键字参数在后                            |

**类比**：法师的魔法书——每个魔法有固定消耗和基础威力。你选魔法（参数 spell_name）、选目标（参数 target）、决定是否加 Buff（参数 boosted）。参数就是你在战斗菜单里选的那几下。默认参数就是"不选就用火球术"的默认选项。

## AI 协作指南

### 场景一：让 AI 帮你把重复代码封装成函数

当你发现同样的逻辑写了两次以上，就该让 AI 帮你"抽函数"：

> "下面三段代码做的事情几乎一样——都是'角色 A 对角色 B 造成伤害'，只是伤害公式略有不同。帮我抽成一个通用函数 `deal_damage(source, target, damage_type='physical')`，如果是 physical 就用 atk-def 公式，如果是 magic 就直接用 source 的 matk 字段。保持保底 1 点伤害的规则。"

把重复代码贴给 AI，它会帮你识别出哪些部分是"不变的框架"（扣 HP、打印日志），哪些是"可变的参数"（伤害类型、计算公式）——这正是函数设计的基本功。

### 场景二：让 AI 帮你设计函数接口

在写函数之前，你可以让 AI 先帮你"设计接口"——也就是决定函数叫什么、要什么参数、返回什么。

> "我要给 RPG 游戏加一个技能系统。技能包括：火焰斩（atk × 1.8 物理伤害，消耗 15 MP）、暴风雪（matk × 2.0 魔法伤害，消耗 25 MP）、治疗术（恢复 50 HP，消耗 15 MP）。请帮我设计一个函数接口——函数叫什么？需要哪些参数？哪些参数有默认值？先给我接口设计，再给完整代码。hero 用字典格式。"

AI 会帮你分析：哪些参数是必须的（caster、target）、哪些有默认值（spell_name）、哪些是可选开关（boosted）。接口设计好了，代码就水到渠成。

## 本章回顾

**一句话总结**：函数是给代码块起名字——就像 RPG 里把"算伤害 + 扣 HP + 播动画"封装成"攻击"指令，一次定义后每次一个函数调用就发动整套动作。

**下章预告**：刘老三把攻击、喝药、施法都封装成了函数——技能栏里整整齐齐。但他发现一个问题：打完一场战斗，系统怎么知道赢了还是输了？"战斗函数干完活，得回来报告一声结果是'胜利'还是'团灭'——不能打完就跑了。"会长翻开魔法书的下一页：**返回值与作用域**——`return` 就是战斗结束后的战报。
