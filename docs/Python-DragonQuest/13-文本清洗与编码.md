## 核心讲解

### 13.1 文本清洗实战——正则去标记

从网页复制攻略是冒险者的日常操作。问题是：复制下来的内容往往带着大量的 HTML 标签，真正的文字被埋在 `<div>`、`<p>`、`<strong>` 里面。

```python
# 从游戏攻略网站抓了 Boss 打法，结果全是 HTML
raw_guide = "<div class='boss'><h2>恶龙攻略</h2><p>弱点：尾巴尖<br/>注意：每3回合释放<strong>龙息</strong></p></div>"

import re
clean = re.sub(r"<[^>]*>", "", raw_guide)
print(clean)
# → "恶龙攻略弱点：尾巴尖注意：每3回合释放龙息"
```

**逐行解读**：

- `import re` —— 引入正则表达式模块（Python 内置的文本处理工具箱）
- `raw_guide` —— 从网页复制下来的原始文本，带着一大堆 HTML 标签
- `r"<[^>]*>"` —— 正则表达式模式：匹配任何以 `<` 开头、`>` 结尾的内容。`[^>]*` 表示「除了 `>` 以外的任意字符，出现任意次数」
- `re.sub(模式, 替换成什么, 在哪个字符串里)` —— 把所有匹配到的 HTML 标签替换成空字符串，也就是直接删掉
- 结果：只剩纯文字内容

**类比**：这好比你在冒险者公会的公告板上揭了一张悬赏令——但上面被人用红笔划了很多线、画了很多圈。你不需要这些标记，你只想知道「在哪打什么怪物、赏金多少」。`re.sub` 就是帮你把所有涂鸦洗掉，只留原文。

再看一个更实用的场景——NPC 的对话文本里混入了情绪标记：

```python
# 清洗 NPC 对话文本（去掉情绪标记）
npc_text = "<anger>你竟敢闯入我的领地！</anger><normal>不过...如果你帮我找一样东西...</normal>"
clean = re.sub(r"<[^>]*>", "", npc_text)
print(clean)
# → "你竟敢闯入我的领地！不过...如果你帮我找一样东西..."
```

**逐行解读**：

- `npc_text` —— 游戏引擎内部的 NPC 对话文本，带情绪标签（`<anger>`、`<normal>`），用于控制 NPC 的表情动画
- 同样的正则模式 `<[^>]*>` —— 匹配所有尖括号包裹的标记
- `re.sub` 替换为空 —— 全部删掉
- 结果 —— 只保留玩家实际看到的对话文字

**冒险思维**：正则表达式是你在文本海洋里的「过滤魔法」。无论水多浑（混了多少 HTML、标记、特殊符号），你都能一键滤出清水（纯文本）。第 12 课我们学了正则的基础语法——`re.findall` 提取数字、`re.match` 检查格式——这一课我们把它用到实战里。

### 13.2 编码——为什么会有乱码

先搞清楚一个核心问题：计算机里所有东西归根结底都是数字。字母 `A` 在计算机内部被存成数字 65，汉字「勇」被存成数字 21191。问题是——哪本密码本规定了 `A=65`？

历史上有很多本「密码本」（编码）：

| 编码 | 覆盖范围 | 典型场景 |
|------|---------|---------|
| ASCII | 英文 + 数字 + 符号（128 个字符） | 最古老的，只认识英文 |
| GBK / GB2312 | 中文 + 英文 | 老中文 Windows 系统 |
| Shift-JIS | 日文 + 英文 | 日本游戏、老式日文文件 |
| UTF-8 | 全球所有文字 | 现代标准，通用 |

当你用编码 A 去读一本用编码 B 写的文件，结果就是——乱码。就像你用中文拼音去读一串日文假名，每个假名都对应了一个错位的中文字。

```python
# 古代石板（日文编码），你用通用解读器（UTF-8）读——乱码
# 换「古代文字解读器」（shift-jis）——就能读了！
# （with open 语法在第14课详细讲解，这里先看 encoding 参数的作用）

with open("古代石板.txt", "r", encoding="shift-jis") as f:
    prophecy = f.read()   # 「当四颗星辰汇聚，勇者将降临」
```

**逐行解读**：

- `open("古代石板.txt", "r", encoding="shift-jis")` —— 打开文件，告诉 Python「这个文件是用 Shift-JIS 编码写的」
- `with ... as f:` —— `with` 语句保证文件用完后自动关闭，不会忘记（第 14 课会详细展开 `with open` 的完整语法和原理）
- `f.read()` —— 读取文件的全部内容
- 如果这里写的是 `encoding="utf-8"` 而文件实际上是 Shift-JIS 编码——你看到的就是一堆 `�` 和乱码

**类比**：考古现场挖出一块石板——上面不是乱刻的，是用古代精灵文字写的。你需要一个懂精灵语的学者来翻译。`encoding` 参数就是告诉 Python「请一位懂 Shift-JIS 的学者来读这个文件」。乱码的本质只有一句话——**写的人用的密码本和读的人用的密码本不一样**。

### 13.3 文件读写中的编码——UTF-8 铁律

既然编码不匹配就会乱码，那该怎么办？答案很简单：**制定一条铁律，所有文件都用同一种编码。**

在现代编程里，这个铁律就是 **UTF-8**。它覆盖全球所有文字——中文、日文、阿拉伯文、甚至 emoji——都能存。一个文件如果在 Windows 上用 UTF-8 存的，拿到 Mac 上、Linux 服务器上，用 UTF-8 读——永远不会乱码。

```python
# 铁律：存档文件统一用 UTF-8
with open("save_data.json", "w", encoding="utf-8") as f:
    f.write(save_json)

# 你的项目规则里写：
# 「所有 open() 都指定 encoding='utf-8'」
# —— AI 帮你记住这个习惯
```

**逐行解读**：

- `open("save_data.json", "w", encoding="utf-8")` —— 以写入模式（`"w"`）打开文件，指定编码为 UTF-8
- `f.write(save_json)` —— 把游戏存档数据写入文件。因为指定了 UTF-8，中文角色名、日文技能名都能正常保存
- 项目规则：把「所有 open() 都指定 encoding='utf-8'」写进你的项目配置文件（比如 CLAUDE.md），这样 AI 助手每次帮你写文件操作代码时，都会自动加上 `encoding='utf-8'`

**类比**：游戏存档格式统一——所有平台、所有版本都用同一个存档格式。不会出现「Windows 存的档，Mac 读不了」的惨剧。UTF-8 就是编程世界的「通用存档格式」。

**关键实践**：

```python
# 读取队友发来的攻略文件——永远指定 encoding
with open("队友的恶龙攻略.txt", "r", encoding="utf-8") as f:
    content = f.read()

# 把自己的冒险日志保存下来——同样指定 encoding
with open("勇者张三的冒险日志.txt", "w", encoding="utf-8") as f:
    f.write("Day 1: 遇到了史莱姆，轻松击败。\n")
    f.write("Day 2: 在遗迹发现古代石板...\n")
```

**逐行解读**：

- 读文件和写文件都显式写了 `encoding="utf-8"` —— 养成习惯，不依赖 Python 的默认行为（不同操作系统默认编码不一样）
- `"r"` 是读模式（read），`"w"` 是写模式（write）——第 14 课会详细讲解所有文件模式
- 写多行内容：每次 `f.write()` 追加内容，`\n` 是换行符
