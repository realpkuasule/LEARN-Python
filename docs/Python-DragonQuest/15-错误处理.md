## 核心讲解

### 15.1 异常是什么——报错不是敌人

先看一个场景：

```python
# 勇者想装备「魔法杖」
hero = {"name": "勇者张三", "class": "战士", "hp": 100, "atk": 15, "def": 5}

# 模拟装备函数
def equip(weapon_name):
    """检查职业匹配并装备"""
    # weapon_db 是武器数据库，这里先定义
    weapon_db = {
        "铁剑": {"atk_bonus": 10, "classes": ["战士", "骑士"]},
        "魔法杖": {"atk_bonus": 5, "mp_bonus": 30, "classes": ["法师", "牧师"]},
    }
    if weapon_name not in weapon_db:
        raise KeyError(f"武器数据库中没有「{weapon_name}」")
    if hero["class"] not in weapon_db[weapon_name]["classes"]:
        raise ClassRestrictionError(f"{hero['class']}职业无法装备{weapon_name}")

# 尝试装备
equip("魔法杖")
# → ClassRestrictionError: 战士职业无法装备魔法杖
```

**逐行解读**：

- `hero` —— 勇者的属性面板，包含职业 `"class": "战士"`。在第 8 课我们定义了统一的攻击函数用到这些属性
- `weapon_db` —— 武器数据库字典。每种武器记录攻击加成（`atk_bonus`）和可用职业列表（`classes`）
- `equip("魔法杖")` —— 勇者尝试装备。函数检查后发现「战士」不在魔法杖的 `classes` 列表里，抛出异常
- 报错信息告诉了你三件事：**什么错了**（`ClassRestrictionError`）、**为什么**（战士不能装备法杖）、**在哪一行**（去那里加判断）

**类比**：你要给战士装备法杖——系统弹出红字「职业不匹配」。不是游戏坏了，是规则不允许。报错 = 系统在告诉你规则。**异常不是 bug，异常是程序在拒绝不合理操作**。

下面是 RPG 开发中最常见的 5 种错误，每一种都对应一个冒险场景：

| 错误类型 | RPG 场景 | 含义 |
|---------|---------|------|
| `KeyError` | 你想用「终极魔法」但这个技能在魔法书里不存在 | 字典里没有这个 key |
| `IndexError` | 背包只有 20 格，你想取第 21 格的东西 | 列表索引越界 |
| `ValueError` | 传送卷轴的坐标写了个负数 | 值类型对但内容不合理 |
| `FileNotFoundError` | 你从没存过档就想读档 | 文件不存在 |
| `TypeError` | 把「火焰斩」当数字来算伤害 | 类型错误 |

看几个具体的例子：

```python
# KeyError —— 魔法书里没有这个技能名
spells = {"火球术": 10, "暴风雪": 25, "治愈术": 15}
# spell_mp = spells["终极魔法"]   # → KeyError: '终极魔法'

# IndexError —— 背包 20 格，取第 21 格
backpack = ["回复药"] * 20
# item = backpack[20]  # → IndexError: list index out of range（0-19 共 20 格，20 是第 21 个）

# FileNotFoundError —— 没存过档就想读
with open("不存在的存档.json", "r", encoding="utf-8") as f:  # → FileNotFoundError
    pass
```

**冒险思维**：当你看到一段报错信息——不要慌。先看最下面一行（错误类型和描述），再看上面几行（在哪出错的）。80% 的错误从错误类型名字就能猜出原因。`KeyError` 就是 key 错了，`FileNotFoundError` 就是文件没找到。

### 15.2 try/except——优雅地兜住错误

既然错误不可避免（用户会乱输入、文件可能不存在、数据可能不完整），我们就要学会「兜住」错误。`try/except` 就是这个兜网。

```python
def equip_weapon(hero, weapon):
    """装备武器——优雅处理各种问题"""
    # weapon_db 武器数据库（必须在使用前定义）
    weapon_db = {
        "铁剑":      {"atk": 10, "classes": ["战士", "骑士"], "level": 1},
        "魔法杖":    {"atk": 5,  "mp": 30, "classes": ["法师", "牧师"], "level": 5},
        "传说之剑":  {"atk": 80, "classes": ["战士", "勇者"], "level": 30},
    }

    try:
        requirements = weapon_db[weapon]["requirements"]
        if hero["level"] < requirements.get("level", 0):
            print(f"等级不足！需要 {requirements['level']} 级")
            return
        if hero["class"] not in requirements.get("classes", [hero["class"]]):
            print(f"职业不匹配！{weapon} 需要 {'/'.join(requirements['classes'])}")
            return
        hero["weapon"] = weapon
        print(f"装备了 {weapon}！攻击力 +{weapon_db[weapon]['atk']}")
    except KeyError:
        print(f"武器数据库里没有「{weapon}」——是不是打错名字了？")
```

**逐行解读**：

- `weapon_db` —— **在函数内部定义武器数据库**，确保它在 `try` 块之前已经存在
- `try:` —— 「尝试执行下面的代码」。如果过程中出了错，不崩溃，跳转到 `except`
- `requirements = weapon_db[weapon]["requirements"]` —— 从数据库获取武器的装备要求。如果玩家输入了一个不存在的武器名（如 `"木棍"`），`weapon_db["木棍"]` 会触发 `KeyError`
- `except KeyError:` —— 「如果捕获到 KeyError 类型的异常，执行这里的代码」。程序不会崩溃，而是打印一个友好的提示
- `hero["level"]` 和 `hero["class"]` —— 在字典里取出勇者的等级和职业，与武器的要求做比较
- `return` —— 条件不满足时提前结束函数，不执行后面的装备逻辑

**类比**：游戏里的装备界面——你点「装备」，系统先检查等级够不够、职业对不对。不够就弹出提示「等级不足」或「职业不匹配」，不会闪退。`try` = 「试试看」，`except` = 「如果失败了，我有备用方案」。整个 `try/except` = 游戏引擎的容错逻辑。

再看一个读档时处理文件不存在的例子：

```python
def safe_load_game(filename="save.json"):
    """安全读档——文件不存在也不闪退"""
    try:
        with open(filename, "r", encoding="utf-8") as f:
            import json
            data = json.load(f)
        print(f"读取存档成功！角色：{data['hero']['name']}")
        return data["hero"]
    except FileNotFoundError:
        print(f"存档文件「{filename}」不存在——开始新游戏吧！")
        return None
    except json.JSONDecodeError:
        print(f"存档文件「{filename}」损坏了——无法读取。")
        return None

# 尝试读档
hero = safe_load_game("save.json")
if hero is None:
    hero = {"name": "勇者张三", "hp": 100, "atk": 15, "def": 5}
    print("已创建新勇者！")
```

**逐行解读**：

- `try:` —— 尝试读档
- `except FileNotFoundError:` —— 文件不存在。这是最常见的场景（新玩家第一次打开游戏）。不崩溃，创建一个新勇者
- `except json.JSONDecodeError:` —— 文件存在但是内容损坏了（比如存到一半停电了）。也优雅处理
- `return None` —— 返回 `None` 表示读档失败，调用者检查后创建新角色
- **一个 `try` 可以有多个 `except`**，分别捕获不同类型的异常——就像 RPG 战斗系统里对不同异常状态用不同的解药

### 15.3 else/finally 与自定义异常

`try/except` 还有两个强力分支：`else` 和 `finally`。

- `else` —— 如果 `try` 块**没有**发生异常，执行这里。相当于「一切顺利，执行庆祝逻辑」
- `finally` —— **无论有没有异常**，都执行这里。相当于「打完不管输赢，战斗日志都保存」

先看一个 RPG 最经典的场景——战斗循环：

```python
# 自定义异常：勇者阵亡
class GameOver(Exception):
    """勇者阵亡——自定义异常"""
    pass

def check_alive(character):
    """检查角色是否还活着——死了就抛出异常"""
    if character["hp"] <= 0:
        raise GameOver(f"{character['name']}倒下了...")

# attack 是第8课定义的统一攻击函数（必须在使用前定义）
def attack(attacker, defender):
    """统一的攻击函数——第8课1节的定义"""
    damage = attacker["atk"] - defender.get("def", 0)
    if damage < 1:
        damage = 1   # 保底1点伤害
    defender["hp"] -= damage
    print(f"{attacker['name']}攻击了{defender['name']}！造成{damage}点伤害")

# 战斗开始——hero 和 monster 必须先定义
hero = {"name": "勇者张三", "hp": 100, "atk": 15, "def": 5}
monster = {"name": "恶龙", "hp": 500, "atk": 80, "def": 30}

try:
    # 战斗主循环
    while hero["hp"] > 0 and monster["hp"] > 0:
        attack(hero, monster)       # 勇者先手（课8.1定义的函数）
        check_alive(monster)        # 怪物倒了吗？
        if monster["hp"] <= 0:
            break
        attack(monster, hero)       # 怪物反击（同样的attack函数！）
        check_alive(hero)           # 勇者倒了吗？
except GameOver as e:
    print(f"\n{e}")
    print("—— GAME OVER ——")
else:
    print("\n战斗胜利！")
finally:
    print("战斗日志已保存。")
    # save_game(hero) ← 第14课学的存档，无论输赢自动存
```

**逐行解读**：

- `class GameOver(Exception):` —— 定义一个自定义异常。继承自 `Exception`，`pass` 表示不需要额外逻辑
- `def check_alive(character):` —— 生命检查函数。HP ≤ 0 时用 `raise` 主动抛出 `GameOver` 异常
- `def attack(attacker, defender):` —— **这是第 8 课第 1 节定义的统一攻击函数**。参数是两个字典（攻击者和防御者），计算 `atk - def` 作为伤害，保底 1 点
- `hero = {...}` 和 `monster = {...}` —— **代码块开始时必须先定义**勇者和怪物的属性面板
- `try:` 包裹整个战斗循环。每个回合：勇者攻击 → 检查怪物 → 怪物攻击 → 检查勇者
- `except GameOver as e:` —— 捕获到 `GameOver` 异常时执行。`as e` 把异常对象赋值给变量 `e`，可以 `print(e)` 显示异常消息
- `else:` —— **只有** `try` 块正常结束（没有触发异常）时才执行。战斗胜利的场景
- `finally:` —— **无论如何**都执行。赢了存档，输了也存档——你不会因为团灭就白打了三小时

**类比**：RPG 的完整战斗流程——每个回合结束都检查「还活着吗」。死了就触发 GameOver 动画（`except`），活到敌人倒下就播放胜利动画（`else`），打完无论输赢都自动存档（`finally`）。`finally` 是最坚固的保险——天塌下来也要执行的代码。

再看一个多层异常处理的实战：

```python
import json

def load_or_create_hero(save_file="save.json"):
    """读档或创建新英雄——多层异常保护"""
    hero = None

    try:
        with open(save_file, "r", encoding="utf-8") as f:
            hero = json.load(f)
    except FileNotFoundError:
        print("没有存档，创建新勇者...")
    except json.JSONDecodeError:
        print("存档损坏，创建新勇者...")
    except PermissionError:
        print("没有读取存档的权限，创建新勇者...")
    else:
        print(f"欢迎回来，{hero['name']}！")
        return hero
    finally:
        print("——初始化和检查完成——")

    # 如果上面没有 return，说明读档失败，创建新勇者
    if hero is None:
        hero = {"name": "勇者张三", "hp": 100, "max_hp": 100,
                "mp": 50, "max_mp": 50, "atk": 15, "def": 5}
    return hero

# 启动！
hero = load_or_create_hero()
print(f"当前 HP：{hero['hp']}")
```

**逐行解读**：

- 三个 `except` 分别处理三种不同的读取失败原因：文件不存在、内容损坏、权限不足。每种情况都有对应的中文提示
- `else` 在读取成功时执行——显示欢迎信息并返回英雄数据
- `finally` 总是执行——打印初始化和检查完成的标志
- 如果 `except` 执行了（读档失败），`hero` 仍是 `None`，后面的 `if hero is None` 创建新角色
- 最终无论如何都返回一个可用的 `hero`——程序不会闪退

## 实战练习

**练习 1（基础）**：写一个除法计算器。要求用户输入两个数字，计算 `a / b`。用 `try/except` 处理以下情况：

1. 用户输入的不是数字（`ValueError`）→ 提示「请输入有效的数字」
2. 除数为 0（`ZeroDivisionError`）→ 提示「除数不能为零」
3. 不管成功或失败，都打印「计算结束」

**练习 2（进阶）**：实现一个带异常处理的背包系统。要求：

1. 定义背包列表 `backpack`，包含若干物品（字符串）
2. 写函数 `use_item(backpack, index)`，根据索引使用物品
3. 用 `try/except` 处理：
   - `IndexError` → 索引越界，提示「背包没有这个位置的物品」
   - `TypeError` → 索引不是整数，提示「请输入数字索引」
4. 用 `else` 打印「使用成功」，用 `finally` 打印当前背包内容
5. 测试：正常使用、越界使用、传入字符串索引

**练习 3（综合）**：完善第 7 课的战斗循环，加入完整的错误处理体系。要求：

1. 定义自定义异常 `BattleEnd`，包含 `winner`（胜者名字）和 `remaining_hp`（胜者剩余 HP）两个属性
2. 定义 `check_hp(hero, monster)` 函数，任一方 HP ≤ 0 时抛出 `BattleEnd`
3. 用 `try/except/else/finally` 包裹整个战斗循环：
   - `except BattleEnd` → 打印胜负结果
   - `else` → 不适用（战斗总是以异常结束），但保留结构
   - `finally` → 打印结算信息（双方剩余 HP）
4. `attack` 函数沿用第 8 课第 1 节的定义
5. 在 `except` 块中，根据 `e.winner` 判断是勇者赢了还是怪物赢了

## 本章回顾

**一句话总结**：异常不是程序的失败——是程序的免疫系统。`try/except` 让程序优雅兜底不崩溃，`else` 在顺利时庆祝，`finally` 让存档（和你三小时的奋战）永远不白费。
